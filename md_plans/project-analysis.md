# 裕行行车记录后视仪项目分析（四天冲刺前置）

## 1. 当前项目状态（基于现状）

- 当前阶段：上位机独立展开中，下位机尚未开始。
- 上下位机通信：尚未编写（MQTT联动未落地）。
- 参考资料状态：除 BSD 盲区检测雷达模块外，其余资料已齐备。
- 时间窗口：4 天冲刺周期，需要以“可联调、可演示、可继续扩展”为核心目标。

## 2. 目标拆解（按依赖顺序）

### 2.1 总目标

在 4 天内形成“上位机可测试 + 下位机可运行基础功能 + MQTT 通信可打通 + 关键模块可集成”的阶段性成果。

### 2.2 依赖链

1. 下位机基础能力（超声波测距、buffer 计数告警）是通信数据源。
2. MQTT 联动依赖上下位机最小可用接口定义。
3. RTOS/WIFI 移植影响下位机任务调度与联网稳定性。
4. BSD 雷达与温湿度火警模块属于功能增强层，需在基础链路稳定后接入。

## 3. 关键风险与应对

### 3.1 风险一：下位机从 0 开始，进度不确定

- 风险说明：若下位机初始化、驱动或定时机制耗时超预期，会压缩通信联调时间。
- 应对策略：优先实现“最小功能闭环”，先保证超声波 + 告警数据可周期输出，再做性能优化。

### 3.2 风险二：MQTT 通信协议边界不清

- 风险说明：上下位机字段命名、主题划分、QoS 选型不一致，会导致联调反复。
- 应对策略：在第 1 天即冻结最小协议草案（topic、payload、心跳、告警）。

### 3.3 风险三：BSD 资料不完整导致集成卡点

- 风险说明：BSD 盲区检测雷达模块信息缺口可能阻塞整体展示。
- 应对策略：将 BSD 作为“可选增量里程碑”，不阻塞主链路验收。

### 3.4 风险四：RTOS/WIFI 移植耦合复杂

- 风险说明：任务调度、驱动中断、网络栈稳定性可能引发隐性问题。
- 应对策略：先完成裸机/简化任务验证，再逐步切换到 RTOS 任务模型。

## 4. 验收导向建议（四天内）

- 最低验收线：
  - 下位机可稳定输出超声波距离与 buffer 告警状态；
  - MQTT 双向控制链路可跑通；
  - 上位机可完成联调测试记录；
  - 至少一个扩展模块（温湿度火警或 BSD 占位）接入流程明确。
- 演示优先级：
  1. 端到端通信与控制；
  2. 关键报警触发；
  3. 扩展模块接入展示。

## 5. 资源与分工建议

- 上位机主线：保持测试与联调工具链持续可用（日志、回放、异常记录）。
- 下位机主线：优先完成可周期上报的数据最小闭环。
- 接口主线：统一 topic、payload 和错误码约定，减少后续返工。

## 6. 输出物建议

本轮四天建议至少形成以下文档输出：

1. 四天执行计划（见同目录 `four-day-project-plan.md`）；
2. MQTT 最小协议约定（后续可单独扩展）；
3. 联调测试记录模板（便于每日复盘）。
